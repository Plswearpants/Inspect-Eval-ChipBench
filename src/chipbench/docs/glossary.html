Glossary

Terms from the Verilog/hardware side and the Inspect eval side that came up while building the ChipBench evaluation — collected here because they weren't obvious from their names.

01

Signals & Verilog Basics

clk
The clock signal — a wire that alternates 0/1 on a steady beat. Every synchronous piece of logic in the design "moves" only at the moment this signal ticks; nothing updates between ticks.
rst_n
Reset, active-low. The trailing _n is a naming convention meaning "this signal does its job when it's 0, not when it's 1" — so rst_n holds the circuit in reset while low, and normal operation only begins once it's pulled high. The plain name reset (no _n) would instead mean active-high — the convention is what tells you which polarity to expect.
posedge / negedge
Short for positive edge / negative edge — the exact instant a signal flips 0→1 (posedge) or 1→0 (negedge). Logic described as "always @(posedge clk)" only reacts at that single instant per clock cycle, not continuously.
port (input / output)
The named wires a module exposes to the outside world — its interface. A module's ports are the only way anything outside it can read or write its signals, the same role a function's parameters and return value play in software.
reg / wire / logic
wire is a plain connection with no memory — it just reflects whatever drives it, instantly. reg can hold a value between clock edges (despite the name, it doesn't always synthesize to a physical register). logic is SystemVerilog's newer, unified type that replaces both and avoids picking wrong.
always block
A block of code describing what happens when a specific event occurs — most commonly always @(posedge clk), meaning "run this logic once per clock tick." It looks like a loop but isn't one; it's re-triggered by the event, not iterated.
assign
A continuous, always-active connection: assign y = a & b; means y is permanently wired to follow a & b, recalculated instantly any time either changes — unlike an always block, which only updates on its triggering event.
8'b0111_0001
Verilog's bit-width literal notation: 8 is the number of bits, b means the digits that follow are binary (d = decimal, h = hex), and underscores are just visual spacers with no meaning — so this is an 8-bit binary value.
X (unknown state)
A special value Verilog signals can hold meaning "not yet determined" — typically before anything has driven them, e.g. immediately at power-on before reset has run. It's neither 0 nor 1; comparing against it needs special handling (see the XOR trick under Verification).
TopModule / RefModule
Naming convention used throughout this dataset's test files, not a Verilog language feature: RefModule is always the golden, known-correct implementation; TopModule is always the slot the model's generated code fills in.
self_contained / not_self_contained / cpu_ip
ChipBench's own dataset category labels, not a Verilog concept. self_contained problems are one complete module; not_self_contained problems instantiate a submodule whose definition lives elsewhere (embedded in the prompt text, not in ref.sv — see submodule staging); cpu_ip is a separate category of real CPU-component designs, harder and unrelated to the self-contained/not-self-contained split.
02

Verification Concepts

testbench
Code that isn't part of the design itself — it exists purely to drive inputs into the circuit being tested and check its outputs. Never synthesized into real hardware; it only exists inside the simulator.
DUT (device under test)
Whichever implementation the testbench is currently judging — here, the model's generated code. Standard verification-industry term, used regardless of whether the "device" is a full chip or a small module.
golden reference / golden model
The trusted, known-correct implementation everything else gets compared against. "Golden" here just means "assumed correct by definition," not any technical property of the code itself.
stimulus
The specific sequence of input values a testbench applies during simulation — the test cases, essentially, whether hand-written (directed) or randomly generated.
differential testing
Checking correctness by running two implementations side by side under identical inputs and comparing their outputs, rather than checking either one against a fixed expected-output list. This is the whole verification strategy ChipBench uses.
functional equivalence
Two pieces of code count as "the same" if they produce identical outputs for every input, regardless of how differently they're written internally — the opposite of comparing source code text.
RTL (register-transfer level)
The abstraction level Verilog is normally written at: describing logic in terms of registers and the transfers between them on each clock edge, one level up from individual transistors/gates.
synthesis vs. simulation
Simulation runs Verilog to check behavior, entirely in software, nothing physical involved. Synthesis instead converts Verilog into an actual gate-level circuit description meant to become real hardware. ChipBench only simulates — nothing here ever gets manufactured.
VCD (waveform dump)
A recorded trace of every signal's value at every moment in a simulation — what a waveform viewer displays as timing diagrams. This is the literal file format embedded in the one-shot debugging prompts.
03

The Simulation Toolchain

Verilog / SystemVerilog
Verilog is the original hardware description language; SystemVerilog is a large superset added later (better types, more verification-oriented constructs like logic). Files ending .sv use the superset; the dataset here mixes both.
iverilog / vvp
Two halves of one tool (Icarus Verilog): iverilog compiles Verilog source into a runnable simulation binary, and vvp executes that binary. Directly analogous to a compile step and a run step.
Verilator
A different kind of simulator: instead of interpreting Verilog directly, it translates it into C++ source code, which then gets compiled and run like any other program. Used here for the refmodel task's cross-language checks.
MODMISSING
Verilator's specific error when a Verilog file instantiates a module by name but no file given to the compiler defines it — e.g. a top-level module calling a submodule whose source was never included in the build. A compile-time failure, unrelated to whether the design's actual logic is correct.
override (C++ keyword)
Marks a method as intentionally replacing a virtual method inherited from a base class — not optional decoration; the compiler checks the signature actually matches something in the base class and errors if it doesn't (e.g. wrong argument list, wrong return type). Used here to catch exactly that: generated CXXRTL code implementing reset()/eval()/commit() with a signature that no longer matches the pinned CXXRTL version's base class.
Yosys
An open-source synthesis tool — normally used to turn Verilog into gate-level circuits. Only its CXXRTL backend gets used in this project, for a different purpose than its main job.
CXXRTL
A Yosys backend that compiles Verilog into standalone C++ classes runnable outside any simulator — effectively turning a Verilog module into a plain C++ object with an eval()-like step function, which is exactly what the reference-model verification needs.
CXXRTL name mangling
CXXRTL's port-naming rule: a Verilog port foo becomes a C++ member p_foo, and every underscore already in the original name gets doubled — data_out becomes p_data__out, not p_data_out. Easy to get wrong since a single-underscore version compiles as a *different*, plausible-looking member name rather than failing loudly — the model has to apply this consistently across every port in a file with no per-port feedback.
wire<N> vs. value<N>
Two different CXXRTL storage types, not interchangeable despite similar names. value<N> is a plain N-bit value — one slot, read and written directly. wire<N> is double-buffered (separate .curr/.next fields plus a .commit() step to latch one into the other) specifically so an edge (0→1 transition) can still be detected after the new value has already been set — needed for the clock signal, since a naive single-slot value would overwrite the "before" state before anything gets a chance to compare it.
SystemC
A C++ class library for modeling hardware behavior directly in C++, without Verilog at all — one of the three reference-model languages in the original paper's benchmark (alongside Python and CXXRTL), though not currently scoreable here since the vendored toolbox has no working SystemC path.
04

Inspect / Eval Terms

Sample
One problem instance — one prompt plus whatever supporting files/target it needs to be scored. A dataset is a list of these.
Task
Inspect's bundle of one dataset + one solver + one scorer (+ optional sandbox) — the unit that inspect eval actually runs.
solver
The component that produces a completion from the model — here, just generate(), meaning "call the model once with the prompt and take its answer," with no tools or multi-turn loop involved.
scorer
The component that judges a completion and assigns it correct/incorrect. In most evals this is a text comparison; here it's the entire compile-and-simulate pipeline covered earlier.
epoch (this context, not ML training)
One full, independent attempt at a sample. Running 20 epochs means asking the model to solve the same problem 20 separate times — nothing to do with training-loop epochs; it exists purely to compute pass@k.
pass@k
The probability that at least one of k independent attempts at a sample succeeds, estimated from a larger number of actual attempts (here, 20) rather than literally sampling exactly k times.
sandbox / image / container
Covered in depth earlier in this conversation — briefly: an image is the frozen, pre-built filesystem template (Ubuntu + the compiled toolchain); a container is a disposable running instance of it, one per sample, torn down after scoring.
staging (dataset construction)
Assembling the actual file contents that get written into a sample's sandbox before it runs — e.g. deciding what ref.sv should contain. Not an Inspect API term, just the name used in this project for the dataset-building step; a bug here (a reference file staged without a submodule it depends on) can guarantee a compile failure regardless of what the model produces, before scoring even considers the model's answer.
