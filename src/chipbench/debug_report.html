ChipBench on Inspect · failure-mode trajectory

Is the score low because the model failed, or because our harness did?

Every Llama 3.1 8B run to date, reclassified sample-by-sample into model-capability failures and pipeline failures — each pipeline failure attributed to the specific bug that caused it. Pass rate alone can't show whether a fix worked; watching a bug's own bucket shrink can.

What's being tested

ChipBench is three separate tasks, each split into sub-categories, each sub-category made of some number of atomic tasks — one independently-scored problem instance (one row in the dataset). Every atomic task below is attempted 20 times (epochs=20) per run to compute pass@1/5/10; a table further down reporting n=900 for refmodel's 45 atomic tasks means exactly that: 45 × 20. The names in this tree (self_contained, cpu_ip, zero_shot, etc.) are the vocabulary the rest of this report uses without re-explaining it.

  • chipbench_verilog_genwrite a Verilog module from a spec 45
    • self_containedone complete module, nothing external needed30
    • not_self_containedthe design depends on a submodule shown separately6
    • cpu_ipreal CPU-component designs (ALU, register file, control unit…)9
  • chipbench_debugfix an injected bug in an existing module 178
    • arithmeticwrong operator or operand in a computation24
    • assignmentwrong signal or wrong blocking/non-blocking form30
    • state_machinewrong state transition or output6
    • timingwrong clock edge or reset polarity29

    89 buggy problems above, each run twice — once zero_shot (spec + buggy code only) and once one_shot (same, plus a VCD waveform trace) — for 178 atomic tasks total.

  • chipbench_refmodelwrite a Python or C++ model behaviorally equivalent to a verilog_gen module 90
    • self_contained × {python, cxxrtl}same 30 problems as above, in both target languages60
    • not_self_contained × {python, cxxrtl}same 6 problems, both languages12
    • cpu_ip × {python, cxxrtl}same 9 problems, both languages18

    Not an independent dataset — every atomic task here reuses one of verilog_gen's 45 problem specs, asking for a reference model in Python or CXXRTL (a C++ hardware-simulation API) instead of Verilog. A third language, SystemC, is part of the paper's original benchmark but isn't scoreable with the vendored toolchain, so it's excluded here.

The trajectory that matters more than pass@1

Four points in time, same 45-problem CXXRTL set, tracking what share of every sample's failure is ours (pipeline) rather than the model's (capability). This is the number we're trying to drive down — not because it makes the pass rate look better, but because every point still in the pipeline band is a sample that never got a fair shot at measuring the model at all.

overall (all 3 categories) cpu_ip self_contained non_self_contained

Y-axis: share of samples classified as pipeline/infrastructure failure (our harness's fault), not model capability. Lower is better and means the category's pass rate is finally trustworthy. self_contained and cpu_ip both reach exactly 0% at stage 5 (their dots are jittered a couple pixels apart at the bottom of the chart purely so both remain visible) — only non_self_contained still has any pipeline share left, and that's entirely the still-open Verilator-strictness gap, not Bug 6.

A controlled before/after, isolated from the rest of the trajectory

The trajectory above is chronological — each stage is whatever bugs happened to be fixed at the time, which is honest history but not a clean experiment. This section is a deliberate two-point comparison instead: same 45 problems, same 20 epochs, same model, both languages, run twice back-to-back. Before: our own faults (Bugs 1 and 2 — the CXXRTL prompt and submodule staging, neither of which the paper's own tooling ever had to get right) are fixed, exactly as they'd stay fixed in any real use of this harness; the paper's own faults (Bugs 3, 5, 6) are deliberately reproduced — Bugs 3 and 6 by temporarily reverting generate_testbench.py/extract_ports.py to their vendored originals and rebuilding the Docker image, Bug 5 via a new stage_missing_defines=False task parameter. After: everything fixed — the same runs already in the trajectory above (refmodel_cxxrtl_postbug6_45 / refmodel_python_postbug6_45). The point: this isolates what the paper's own bugs cost, on top of a harness that's already doing its own part correctly — not conflated with our own porting mistakes.

This report is Llama-only, but the fix itself isn't Llama-specific. GPT-3.5 Turbo's refmodel runs against this same fully-fixed pipeline show the identical signature in both languages independently: self_contained/cpu_ip at 0% pipeline share, only verilator_rejects_construct left in not_self_contained (10/450 in each language, all one problem). The six bugs were all found and fixed using Llama's runs alone — this is the check for whether those fixes happened to be tuned to Llama's specific failure patterns. Four for four (2 models × 2 languages), they weren't. Full numbers in summary.md.

Why some bug counts go up when you fix something else

Three times in this project, fixing one bug didn't reduce the total failure count — it revealed a second bug that had been hiding underneath the first the entire time, because the compiler only reports the first error it hits per sample. Reading a bucket's count rising without this context looks like a regression; it isn't.

Method

Every run loaded via Inspect's own inspect_ai.analysis.samples_df() — id, metadata, score value, and full score explanation across one or many .eval logs in a single call, no hand-rolled log parsing. Classification on top is in agent_artefacts/chipbench/failure_analysis/classify.py: every pattern was derived from reading actual score explanations, not guessed in advance, and unclassified is a real bucket — nothing gets silently forced into capability or pipeline just to make the categories add up. bug_trajectory.py re-applies this classifier to all fourteen runs above; the underlying counts are in bug_trajectory.json alongside it. Every run's identity (raw .eval log, renamed json export, model, stage) is indexed once in run_meta.py/run_meta.md — that's the file to check for what any given log actually was, including runs not part of this trajectory (other models, blocked attempts, calibration runs).

Two disambiguations needed more than a keyword match to get right, both documented in code: a p_clk-missing error is Bug 3 only if the circuit is actually combinational (checked against each problem's own ref.sv, not assumed); a redeclaration error is Bug 6 only if it's anchored to testbench.cpp, not the model's own generated.cc — an early, looser version of this check produced false positives that were caught by re-reading the raw text behind a surprising number.
